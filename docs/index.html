1. Introduction

Welcome to the documentation for the CubeXell CubeSolver. This project is a complete hardware and software pipeline designed to autonomously scan, calculate, and physically solve a 3x3 Rubik's Cube.

System Overview

The system operates in a sequential pipeline:

  1. Camera & Vision: An overhead Raspberry Pi camera captures the cube's faces. OpenCV processes the images to extract sticker colors.
  2. State Generation: The color data is mapped into a standard mathematical representation of a Rubik's Cube.
  3. Solver Engine: An algorithm (e.g., Kociemba, CFOP, or Beginner) calculates the most efficient series of moves to solve the cube.
  4. Motor Control: The solution string is translated into electronic signals sent to stepper motor drivers, physically rotating the cube's faces to the solved state.

Hardware Requirements

  • Raspberry Pi (3B+ or 4 recommended)
  • Raspberry Pi Camera Module v2
  • 6x NEMA 17 Stepper Motors (one for each face)
  • 6x A4988 or DRV8825 Motor Drivers
  • Custom 3D printed frame and cube center-caps

2. Installation

Follow these steps to set up the software environment on your Raspberry Pi.

Prerequisites

Ensure your Raspberry Pi is running a recent version of Raspberry Pi OS with Python 3.9+ installed. You must also enable the camera interface via sudo raspi-config.

Install Dependencies

sudo apt update
sudo apt install python3-pip libopencv-dev python3-opencv

# Clone the repository
git clone https://github.com/randomcuber/cubesolverCampFull
cd cubesolverCampFull

# Install Python requirements
pip3 install -r requirements.txt

Verify Setup

Run the test script to ensure the camera and GPIO pins are accessible:

python3 main.py --test-hardware

3. Reference

This section documents the core classes that make up the CubeSolver architecture. Understanding these classes is crucial if you intend to modify the motor logic or vision system.

The MotorController handles the GPIO signals sent to the stepper motor drivers. It translates standard cube notation (U, R', D2) into step and direction pulses.

Responsibilities:

Classes: Solver & Cube

The Cube class maintains the virtual state of the puzzle. The Solver class takes a Cube instance and returns an array of moves.

from core import Cube, Solver

my_cube = Cube(state_string="DUUBULDBFRBFRRULLLBRDFFFBLURDBFDFDRFRULCDUUB... ")
my_solver = Solver(method="kociemba")

solution = my_solver.solve(my_cube)
print(solution) # Output: ["R", "U", "R'", "F2", "D'"]

4. Usage

The primary way to interact with the system is through the command line using main.py.

Running a Full Solve Pipeline

To run the default sequence (Scan -> Calculate -> Solve physically):

python3 main.py --run

Command Line Arguments

You can customize the execution using flags:

  • --method [name]: Choose the solving algorithm (kociemba, cfop, beginner).
  • --dry-run: Scans and calculates the solution, but does not engage the motors.
  • --scramble [length]: Instructs the motors to randomly scramble the cube before solving.
python3 main.py --run --method cfop --dry-run

Configuration File

Persistent settings are stored in config.json in the root directory. You can edit this file to change default behaviors without using command line flags.

{
    "default_solver": "kociemba",
    "motor_speed_rpm": 120,
    "camera_resolution": [640, 480],
    "log_level": "INFO"
}

5. Tuning

Because lighting and hardware builds vary, the vision and motor systems must be tuned to your specific physical setup.

Tuning Vision Coordinates

The camera needs to know exactly which pixels to look at to determine a sticker's color. These are defined as an array of (x, y) coordinates in the OpenCVRaspberryPiCamera class.

To tune these, run the calibration tool:

python3 main.py --calibrate-vision

This will open a GUI window showing the live camera feed. You will see green circles drawn on the frame using OpenCV.

# Inside OpenCVRaspberryPiCamera.py
# Adjust these coordinates to match the center of your cube's stickers in the frame
self.sticker_coords = [
    (120, 120), (320, 120), (520, 120), # Top row
    (120, 240), (320, 240), (520, 240), # Middle row
    (120, 360), (320, 360), (520, 360)  # Bottom row
]

# Drawing logic for visual debugging
for (x, y) in self.sticker_coords:
    cv2.circle(frame, (x, y), radius=5, color=(0, 255, 0), thickness=-1)

If the circles do not land perfectly in the center of your physical cube's stickers, adjust the integers in self.sticker_coords until they do.

Tuning Motor Timing

If your stepper motors are skipping steps or binding, you may need to adjust the delay between steps. Lower delays mean faster turns, but less torque. Open motor_config.py and adjust the STEP_DELAY.

# Increase this value if motors skip steps (e.g. 0.002)
# Decrease for faster solves (e.g. 0.0008)
STEP_DELAY = 0.001

6. Contributing

We welcome pull requests! The two most common areas for contribution are adding new mathematical solvers or adding support for different motor driver boards.

Adding a New Solving Method

To add a custom solver, create a new class in the solvers/ directory that inherits from BaseSolver. You must implement the solve() method.

from core.base_solver import BaseSolver

class RouxSolver(BaseSolver):
    def __init__(self):
        super().__init__(name="roux")

    def solve(self, cube_state_string):
        """
        Input: 54-char string representing the cube.
        Output: List of strings representing moves.
        """
        solution_moves = []
        # Implement Roux logic here
        return solution_moves

Once created, register it in the solver_factory.py file so it can be called via the command line.

Supporting New Hardware

If you are using an Arduino or a different motor shield instead of raw GPIO pins, you can write a custom wrapper. Create a class that implements the MotorController interface (specifically the rotate_face method) and submit it via a Pull Request.

Steps to Contribute:

  1. Fork the repository.
  2. Create a feature branch (git checkout -b feature/arduino-support).
  3. Commit your changes (git commit -m "Add Arduino serial motor controller").
  4. Push to the branch (git push origin feature/arduino-support).
  5. Open a Pull Request on GitHub.
